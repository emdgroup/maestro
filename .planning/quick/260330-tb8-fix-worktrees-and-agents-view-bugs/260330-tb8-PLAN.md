**Bug 1 — Cleanup modal doesn't close after delete:**

The AlertDialog in the worktree detail header is uncontrolled (no `open` prop). When `deleteMutation.mutate` succeeds and calls `onSelect(null)`, the selectedWorktree becomes null, which unmounts the detail panel. **Part A — Backend: Add delete_execution_log IPC command**

In `src-tauri/src/ipc/execution_handlers.rs`, add a new IPC command after `list_executions_with_task_info`:

```rust
#[tauri::command]
#[specta::specta]
pub async fn delete_execution_log(
    app_state: State<'_, Arc<AppState>>,
    execution_id: i32,
) -> Result<(), String> {
    // First, clean up PTY session if it exists
    let mut sessions = app_state.pty_sessions.lock().await;
    sessions.remove(&execution_id);
    drop(sessions);

    // Delete from database
    let conn = app_state.db.lock().map_err(|e| format!("Lock failed: {}", e))?;
    conn.execute(
        "DELETE FROM execution_logs WHERE id = ?",
        rusqlite::params![execution_id],
    )
    .map_err(|e| format!("Failed to delete execution log: {}", e))?;

    println!("delete_execution_log: deleted execution {}", execution_id);
    Ok(())
}
```

Register in `src-tauri/src/lib.rs` — add `crate::ipc::delete_execution_log` to the `collect_commands!` macro in `create_builder()`.

After adding the Rust command, run `pnpm tauri:gen` to regenerate TypeScript bindings so `commands.deleteExecutionLog` becomes available.

export function useDeleteExecutionMutation() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: async ({ executionId }: { executionId: number }) => {
      return await api.deleteExecutionLog(executionId);
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: executionQueryKeys.all });
      toast.success("Session deleted");
    },
    onError: (error) => {
      toast.error(`Failed to delete session: ${error}`);
    },
  });
}

**Part E — Fix interactive session branch_name in query**

The `list_executions_with_task_info` SQL query joins worktrees via `w.task_id = el.task_id`. For interactive sessions where `task_id IS NULL`, this join produces NULL for `branch_name`. Fix the query to also join worktrees by branch for interactive sessions:

Replace the LEFT JOIN on worktrees with:
```sql
LEFT JOIN worktrees w ON (w.task_id = el.task_id AND el.task_id IS NOT NULL)

        SELECT w2.branch_name FROM worktrees w2
        WHERE w2.project_id = ?1 AND w2.task_id IS NULL
        ORDER BY w2.created_at DESC LIMIT 1

```

Actually this is too complex. A simpler approach: store `branch_name` directly on `execution_logs` for interactive sessions. But that requires a schema change.

Simpler fix: In `spawn_interactive_execution`, update the execution log row to include the branch name in the terminal_output or a comment field. Actually even simpler — just add the branch_name info to the WHERE clause. Let me reconsider.

The simplest fix: In `list_executions_with_task_info`, for interactive sessions, get branch_name from the worktree that was created without a task_id. Change the LEFT JOIN to:
