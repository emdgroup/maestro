# Requirements: Maestro v1.3 ‚Äî Agents & Worktrees

**Status:** Draft (2026-03-29)
**Milestone:** v1.3
**Source:** User brainstorm + 4-agent research pass

---

## Scope

Replace both placeholder views (`AgentMonitor`, `WorktreeManager`) with fully functional, real-data-backed screens. Includes a backend overhaul from pool-based to on-demand worktree creation.

## Out of Scope (v1.3)

- Multiple terminals side-by-side or tab strips (sidebar row IS the tab selector)
- Git operations (merge/push/rebase) in the Worktrees view (review flow owns this)
- Real-time git status polling on background interval (refresh on view open is sufficient)
- Agent scheduling or queuing from the Agents view (Kanban is the dispatcher)
- Batch zombie cleanup IPC (manual delete per-card is sufficient for now)
- Uncommitted file count inline on worktree card (too expensive at list time)
- `@xterm/addon-search` for live in-terminal search (defer to v2)

## Requirements

### Backend Overhaul (Phase 25)

**REQ-01** Schema v3 migration ‚Äî drop worktrees table and recreate with new schema (drop-and-recreate strategy; no production data exists). New schema: `id`, `project_id`, `task_id` (FK, nullable for manually created worktrees), `branch_name`, `path`, `git_status`, `created_at`.

**REQ-02** Worktree model overhaul ‚Äî remove `WorktreeStatus` pool enum (`Available`, `Leased`, `InUse`, `Dirty`). Remove `PoolStatus` struct. Add `WorktreeWithStatus` view model (worktree + task info + agent status + is_zombie flag). Add `ExecutionWithTask` view model (execution_log + task name + worktree branch).

**REQ-03** Real `git worktree add/remove` implementation ‚Äî the local stubs in `git/mod.rs` (currently TODO) must be implemented using `tokio::process::Command`. Pattern matches existing remote implementations in `git/remote.rs`.

**REQ-04** On-demand worktree path convention ‚Äî worktrees created at `.maestro/worktrees/task-{task_id}` inside the project root. This path is a named constant in the Rust backend.

**REQ-05** Remove 5 pool IPC commands ‚Äî `initialize_worktree_pool`, `lease_worktree`, `return_worktree`, `get_pool_status`, `recover_dirty_worktrees` removed from `worktree_handlers.rs` and `lib.rs` registration.

**REQ-06** Add `list_worktrees_with_status` IPC ‚Äî returns `WorktreeWithStatus[]` for a project; runs `git worktree list --porcelain` to get real worktree paths, enriches with DB task links and agent execution status via SQL join. `git_status` (uncommitted changes) computed per-query via `git status --porcelain` per worktree; not cached.

**REQ-07** Add `get_worktree_diff` IPC ‚Äî returns unified diff string for a worktree vs its origin branch; piped directly into `@git-diff-view/react` format.

**REQ-08** Add `create_worktree` IPC ‚Äî creates a new worktree at the specified path with the specified branch; used by both the on-demand agent spawn path and the manual creation dialog in the Worktrees view.

**REQ-09** Add `delete_worktree` IPC ‚Äî removes worktree directory and DB row; calls `git worktree remove` then deletes the DB entry.

**REQ-10** Add `list_executions_with_task_info` IPC ‚Äî returns `ExecutionWithTask[]` for a project, joining `execution_logs` with `tasks` and `worktrees`; sorted descending by `started_at`. Used by Agents view sidebar.

**REQ-11** `spawn_agent_execution` uses on-demand create ‚Äî replace `lease_worktree()` call at spawn time with `create_worktree_for_task(task_id)` that creates a real worktree at `.maestro/worktrees/task-{id}`.

**REQ-12** Finalization blocks delete not return ‚Äî both `spawn_agent_execution` and `resume_agent_execution` finalization closures must delete the worktree (not set `status = 'Available'`). All `status = 'Available'` writes in `execution_handlers.rs` audited and removed.

**REQ-13** No blocking git subprocess in async IPC ‚Äî all git subprocess calls in async IPC handlers use `tokio::process::Command`; `git2` crate synchronous calls wrapped in `tokio::task::spawn_blocking`.

**REQ-14** New Rust crates ‚Äî `git2 = { version = "0.20.4", features = ["vendored-libgit2"] }` and `notify = "8.2.0"` added to `Cargo.toml`.

**REQ-15** TypeScript bindings regenerated ‚Äî `pnpm tauri:gen` run after all Rust model changes; `bindings.ts` reflects new model shapes before any frontend work begins.

### Agents View (Phase 26)

**REQ-16** `useExecutionsWithTaskInfoQuery(projectId)` hook ‚Äî added to `execution.service.ts`; 2-second refetch interval; returns `ExecutionWithTask[]`.

**REQ-17** `AgentMonitor.tsx` rewritten ‚Äî real sidebar list from `ExecutionWithTask[]`; no static placeholder data. Each row shows: status dot (color-coded), task name, status label + elapsed time, worktree branch name.

**REQ-18** Mixed list (active + history) ‚Äî sidebar shows all executions sorted by `started_at` descending; no separate tabs for active vs history.

**REQ-19** Status filter toolbar ‚Äî filter chips for All / Running / Done / Failed; task name search input. Both applied client-side to the full list.

**REQ-20** Live xterm.js terminal ‚Äî clicking any sidebar row mounts `ExecutionTerminal` keyed by `task_id`; Tauri channel detached from previous selection and attached to new selection.

**REQ-21** Dead session handling ‚Äî for non-Running executions, do NOT call `attach_terminal`; instead fetch `terminal_output` from the DB via execution record and write it directly to xterm.js via `terminal.write(history)`. Show "Session ended" notice above terminal.

**REQ-22** xterm.js lifecycle ‚Äî `useEffect` cleanup always calls `terminal.dispose()`, `fitAddon` cleanup, `ResizeObserver.disconnect()`, and `detach_terminal` IPC on unmount or selection change.

**REQ-23** Deep link auto-select ‚Äî `pendingAgentId` from `navigationStore` used for initial selection on view mount; fallback to most-recent running execution if no pending ID.

**REQ-24** `AgentsView.tsx` wired ‚Äî passes `ExecutionWithTask[]` from TanStack Query as props to `AgentMonitor`; no direct IPC calls inside `AgentMonitor`.

### Worktrees View (Phase 27)

**REQ-25** `worktree.service.ts` created ‚Äî new service with `useWorktreesQuery(projectId)`, `useWorktreeDiffQuery(worktreeId)`, `useDeleteWorktreeMutation`, `useCreateWorktreeMutation`.

**REQ-26** `WorktreeManager.tsx` rewritten ‚Äî card grid from `WorktreeWithStatus[]`; no static placeholder data.

**REQ-27** Worktree card content ‚Äî each card shows: branch name, linked task name (or "No task" if manually created), agent status badge (active/idle), last activity timestamp.

**REQ-28** Zombie/prunable badge ‚Äî if `is_zombie: true` (no task link + no recent activity), card shows a "Zombie" badge. Never auto-delete zombies; badge is informational only.

**REQ-29** Right panel detail ‚Äî clicking a worktree card opens a right panel (same pattern as BacklogView + BacklogTaskSheet). Panel shows: worktree metadata + git diff vs origin branch via `@git-diff-view/react`.

**REQ-30** Manual "Clean up" action ‚Äî per-card button (visible on zombie cards, available on all cards); triggers confirmation dialog before calling `delete_worktree`.

**REQ-31** Task deep link ‚Äî clicking task name on a worktree card navigates to Kanban view + highlights the task via `navigationStore.navigate({ taskId })`.

**REQ-32** Manual worktree creation ‚Äî "New Worktree" button in view header opens a dialog: branch name input + path input (no task link). Calls `create_worktree` IPC. Newly created worktrees appear in the list immediately via query invalidation.

**REQ-33** `WorktreesView.tsx` wired ‚Äî passes `WorktreeWithStatus[]` from TanStack Query as props to `WorktreeManager`; no direct IPC calls inside the display component.

---

### Zombie Cleanup on Project Open (Phase 28)

**REQ-34** `cleanup_zombie_worktrees(project_id)` IPC command ‚Äî finds worktrees where: `task_id IS NULL` OR task status is Done/Archived, AND path exists on disk (confirmed by `git worktree list`). Never deletes based on DB state alone.

**REQ-35** Time threshold ‚Äî only considers worktrees with `created_at` older than 10 minutes to avoid false positives on actively starting agents.

**REQ-36** Called on project open ‚Äî replaces `recover_dirty_worktrees` call in `App.tsx` `useEffect` on project load.

## Decisions Captured

| Decision | Answer |
|----------|--------|
| Worktree path convention | `.maestro/worktrees/task-{id}` inside project root |
| `git_status` computation | Per-query (on Worktrees view open); not cached |
| Schema v3 migration | Drop-and-recreate (no production data) |
| Manual worktree creation | Included (branch + path, no task link) |
| Agent list format | Mixed active + history, sorted by time descending |
| Dead session behavior | Render DB `terminal_output`, show "Session ended" notice |
| Zombie detection | Informational badge + manual cleanup; never auto-delete |
| Worktree lifecycle | On-demand create at task start; delete on completion/merge |
