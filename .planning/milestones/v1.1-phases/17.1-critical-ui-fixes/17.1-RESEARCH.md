**Researched:** 2026-02-11

## Summary

Phase 17.1 addresses urgent production-blocking issues that weren't caught during Phase 17's "paper-only" verification:

1. **Production mode folder selection is broken** - User can select a folder via native dialog, but the app never transitions to the main UI (ProjectPicker component loads but doesn't complete the selection flow)

2. **UI lacks modern aesthetic** - Despite having Tailwind CSS, shadcn/ui components, and theme system in place, the app looks unstyled and unpolished. The exemple/ reference shows proper UX patterns (inline project dropdown, tab navigation, compact header)

3. **Missing critical UX patterns** - The current UI uses a separate full-screen ProjectPicker stage instead of inline project selection in the header; navigation lacks tab icons and compact layout; Kanban board lacks visual refinement

4. **System accent color not integrated** - Designed in Phase 14-15 but never actually applied from system settings

5. **No actual visual testing performed** - Phase 17 claims accessibility and responsive design success, but all verification was "paper-only" (reading code). Need Playwright visual regression tests to validate actual rendering.

**Primary recommendation:** Debug production folder selection flow, implement header-based project dropdown and tab navigation matching exemple/ patterns, inject system accent color properly, add Playwright visual testing to prevent future false claims.

## Architecture Patterns

### Pattern 1: Production Build Debugging (ProjectPicker Issue)

**What:** The folder selection flow has three stages (select → local → folder picked) but in production mode, after folder selection via native dialog, the app doesn't call `onProjectSelected()` or the call fails silently.

**Root causes to investigate:**
1. Tauri IPC invoke calls failing in production (security context)
2. Dialog plugin returning undefined/null for selected path
3. Settings save failing and blocking flow continuation
4. Error being swallowed without console logging

**Example debugging approach:**
```typescript
// Add production diagnostics to ProjectPicker
async function handleFolderPicker() {
  try {
    const selectedPath = await openDialog({
      directory: true,
      multiple: false,
      title: "Select Project Directory",
    });

    if (!selectedPath) {
      console.warn("[DEBUG] Dialog returned no path");
      return; // Don't proceed if no selection
    }

    console.log("[DEBUG] Dialog path:", selectedPath);
    await onProjectSelected(selectedPath);
    console.log("[DEBUG] onProjectSelected completed");
  } catch (err) {
    console.error("[DEBUG] Dialog or selection failed:", err);
    setError(String(err));
  }
}
```

**When to use:** Every IPC call in production should log success/failure to console for debugging. Add logging at points where flow can break.

### Pattern 3: System Accent Color Integration

**What:** CSS variables already defined (--accent: 217 91% 35%), but system accent color from OS never injected into page.

**Current state:**
- Tailwind config references CSS variables ✓
- index.css defines fallback colors ✓
- ThemeProvider manages dark mode ✓
- **Missing:** Reading system accent color from Tauri and injecting into CSS variables

**Implementation approach:**
```typescript
// ThemeProvider.tsx - add system accent color loading
useEffect(() => {
  async function loadSystemAccentColor() {
    try {
      // Tauri doesn't provide native accent color API
      // Option 1: Use system theme colors via prefers-color-scheme
      // Option 2: Add Tauri plugin for accent color
      // Option 3: Use fallback HSL values and allow user override

      // For now: detect from CSS (light mode = 217 91% 35%, dark = 217 91% 71%)
      const isDark = window.matchMedia("(prefers-color-scheme: dark)").matches;
      const accentColor = isDark ? "217 91% 71%" : "217 91% 35%";
      document.documentElement.style.setProperty("--accent", accentColor);
    } catch (err) {
      console.error("Failed to load system accent color:", err);
    }
  }

  loadSystemAccentColor();
}, []);
```

**Real integration (Phase 14-15 planned but not executed):**
- Tauri doesn't provide system accent color API natively
- Could use platform-specific calls (WinAPI for Windows, NSColor for macOS, XDG for Linux)
- For v1.1: Keep fallback approach, allow user override in settings

### Pattern 4: Playwright Visual Testing

**What:** Create automated visual regression tests to catch UI breakage before claiming success.

**Test structure:**
```typescript
// tests/e2e/visual.spec.ts
import { test, expect } from '@playwright/test';

test.describe('Visual Regression Tests', () => {
  test('ProjectPicker renders correctly in light mode', async ({ page }) => {
    await page.goto('http://localhost:5173');

    // Wait for ProjectPicker to load
    await page.waitForSelector('.project-picker');

    // Compare against baseline screenshot
    await expect(page).toHaveScreenshot('projectpicker-light.png');
  });

  test('AppHeader with project dropdown in dark mode', async ({ page }) => {
    await page.evaluate(() => {
      document.documentElement.classList.add('dark');
    });
    await page.goto('http://localhost:5173');

    // Switch to kanban view (after project selected)
    await page.click('[data-view="kanban"]');

    // Take screenshot of header
    const header = page.locator('header');
    await expect(header).toHaveScreenshot('header-dark.png');
  });

  test('Production build renders without layout shifts', async ({ page }) => {
    // Test built version from dist/
    await page.goto('file://' + process.cwd() + '/dist/index.html');
    await page.waitForLoadState('networkidle');

    // Check for CLS (Cumulative Layout Shift)
    const cls = await page.evaluate(() => {
      return new Promise(resolve => {
        new PerformanceObserver((list) => {
          for (const entry of list.getEntries()) {
            if (entry.hadRecentInput) continue;
            resolve(entry.value);
          }
        }).observe({ type: 'layout-shift', buffered: true });
        setTimeout(() => resolve(0), 1000);
      });
    });

    expect(cls).toBeLessThan(0.1); // WCAG passes < 0.1
  });
});
```

**Source:** Playwright official docs - https://playwright.dev/docs/test-snapshots

**When to use:** Visual regression tests after any UI changes, to prevent "works on paper" bugs from reaching production.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|------------|-------------|-----|
| Folder selection (native dialog) | Custom file browser in Electron | @tauri-apps/plugin-dialog | Tauri handles platform-specific dialogs securely |
| UI component library | Custom React components from scratch | shadcn/ui (already installed) | 11 components ready, better accessibility, theme integration |
| Visual regression testing | Manual screenshot comparison | Playwright snapshots + CI | Automated, deterministic, catches subtle shifts |
| Theming system | localStorage + manual theme class toggle | ThemeProvider (already in code) + system-first detection | Better UX, respects OS preference, prevents FOUC |
| Header layout | Custom flexbox layout | Tailwind utilities (flex, gap, px, py) | Faster, consistent, responsive by default |

**Key insight:** UI quality issues often come from incomplete integration of existing tools (Tailwind, shadcn, ThemeProvider exist but aren't fully wired). Don't add new tools—complete what's already there.

## Common Pitfalls

### Pitfall 1: Tauri IPC Production Silent Failures

**What goes wrong:** IPC calls work in dev mode but fail silently in production because:
- Tauri security context is stricter in built binaries
- Plugin must be explicitly enabled in tauri.conf.json
- Errors are not logged to browser console in production

**Why it happens:** Production builds disable source maps and verbose logging; Tauri security sandboxing is more restrictive than dev mode.

**How to avoid:**
1. Always wrap IPC calls in try-catch with explicit logging
2. Test actual production build (pnpm tauri build) locally before claiming success
3. Enable Tauri logging in production: check tauri.conf.json for "debug" config
4. Verify plugins are enabled: check tauri.conf.json features array

**Warning signs:**
- "Works on localhost:5173 but not in built app"
- No error message, app just hangs or doesn't respond
- Dialog plugin return value is null/undefined

### Pitfall 2: "Paper-Only" Verification Without Actual Testing

**What goes wrong:** Phase 17 claimed WCAG AA compliance and responsive design by reading code without opening the app. Real issues caught by Phase 17.1:
- Production build folder selection broken (no code review would catch this)
- UI looks unstyled despite Tailwind being installed (parsing CSS wouldn't catch this)
- Theme system partially broken (requires runtime testing)

**Why it happens:** Verification is faster on paper, tempting shortcut when under time pressure. Code presence doesn't mean feature works end-to-end.

**How to avoid:**
- NEVER mark Phase success without actually running the app
- For UI: open in dev mode (pnpm dev), actually use the features
- For production: build (pnpm tauri build) and test the binary
- For accessibility: use browser DevTools (F12 → Accessibility tab) or wave.webaim.org
- Screenshot features to verify visual quality matches requirements

**Warning signs:**
- "Based on code review" instead of "Tested and verified"
- Success criteria marked green without evidence of actually running feature
- Verification completed suspiciously fast (UI work should take time)

### Pitfall 3: CSS Classes Purged in Production Build

**What goes wrong:** Tailwind purges unused CSS classes during production build, causing layout to break. Example: grid-cols-5 not in CSS because KanbanBoard.tsx wasn't scanned.

**Why it happens:** Tailwind's JIT mode only includes classes found in content paths during build. If template files aren't in content paths or are imported dynamically, classes get purged.

**How to avoid:**
1. Verify content paths in tailwind.config.ts include all template files
2. Use bundle verification script (already exists: scripts/verify-bundle.mjs)
3. Don't generate class names dynamically: `className={'grid-cols-' + numCols}` gets purged
4. Always test production build locally: pnpm build && pnpm preview

**Warning signs:**
- Layout works in dev (pnpm dev) but breaks after build
- CSS bundle size suspiciously small
- "unexpected behavior in production only"

### Pitfall 4: Theme System Not Applied on Startup

**What goes wrong:** Dark mode flicker (FOUC) on startup, or theme preference not loading from settings.

**Why it happens:** ThemeProvider loads after React renders, so if theme isn't injected before first paint, user sees light mode flash before it switches to dark.

**How to avoid:**
1. Load theme before React renders (main.tsx or index.html preload script)
2. Inject 'dark' class on html element immediately
3. Test with hard refresh (Ctrl+Shift+R) to clear cache and verify startup
4. Check ThemeProvider loads theme from database on mount

**Warning signs:**
- "I see a flash of light mode when app starts" (even if you prefer dark)
- Dark mode works after opening settings, but not on fresh app start
- Theme reset on every app restart

### Pitfall 5: Production Folder Selection State Not Persisting

**What goes wrong:** User selects folder, modal closes, but app shows ProjectPicker again on next open. Settings didn't save.

**Why it happens:**
- invoke("save_settings") failed but didn't throw
- Database transaction rolled back
- Settings loaded before save completed

**How to avoid:**
1. Always await invoke() calls, don't fire-and-forget
2. Add error logging to every invoke() call
3. Verify settings were actually saved by reading them back
4. Use database.open() with proper error handling in Rust

**Warning signs:**
- "I select a project but it doesn't remember"
- Same ProjectPicker shows every time I restart
- No error in console, but settings didn't persist

## Code Examples

### Example 1: Production-Safe IPC Wrapper

```typescript
// src/lib/tauri-safe.ts
import { invoke } from "@tauri-apps/api/core";

/**
 * Wraps Tauri invoke with error logging for production debugging
 */
export async function safeInvoke<T = unknown>(
  command: string,
  args?: Record<string, unknown>
): Promise<T | null> {
  try {
    console.log(`[Tauri] Invoking ${command} with args:`, args);
    const result = await invoke<T>(command, args);
    console.log(`[Tauri] ${command} succeeded:`, result);
    return result;
  } catch (err) {
    console.error(`[Tauri] ${command} failed:`, err);
    throw err;
  }
}

// Usage in ProjectPicker
const project = await safeInvoke<Project>("get_or_create_project", {
  path: projectPath,
});
```

## Open Questions

1. **System accent color API on Tauri:**
   - What we know: Tauri v2 doesn't provide native system accent color API
   - What's unclear: Should we use WinAPI/NSColor/XDG for each platform, or keep fallback HSL values?
   - Recommendation: For v1.1, keep current fallback (217 91% 35%) and allow user override in settings. Real platform integration could be future work.

2. **Production folder selection failure root cause:**
   - What we know: Works in dev mode but fails in production builds
   - What's unclear: Is it a Tauri security context issue, plugin configuration issue, or settings persistence issue?
   - Recommendation: Add comprehensive logging to all IPC calls, build production binary locally, test with logged console output.

3. **Playwright headless browser in CI/CD:**
   - What we know: Playwright can run visual tests against dev server or built app
   - What's unclear: Should tests run against dev server (faster, but less realistic) or production build (slower, more realistic)?
   - Recommendation: Run against production build in CI to catch real issues; use dev server for local development iteration.

4. **Performance metrics for visual regression:**
   - What we know: CLS (Cumulative Layout Shift) should be < 0.1 for WCAG compliance
   - What's unclear: Should we also track LCP (Largest Contentful Paint) and FID (First Input Delay)?
   - Recommendation: For v1.1, focus on CLS (layout stability). LCP/FID can be optimized in v1.2 performance phase.
