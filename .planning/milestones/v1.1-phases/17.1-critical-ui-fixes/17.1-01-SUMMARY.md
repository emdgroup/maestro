# Phase 17.1 Plan 01: Production-Safe IPC Logging

## Summary

Fixed production mode folder selection flow by implementing comprehensive IPC logging and error handling. Created a production-safe wrapper around Tauri's invoke command that logs all IPC communication to browser console, making production builds fully debuggable without requiring source maps or debug builds.

**One-liner:** Production-safe IPC wrapper with comprehensive logging enables full production build debugging via browser console

## Objective

Fix production mode folder selection blocker by adding comprehensive logging and error handling to all IPC calls. The original issue: users could select a folder but the app didn't proceed to main UI in production builds. Root cause was lack of visibility into IPC failures.

## Execution

### Task 1: Create production-safe IPC wrapper with logging (Commit: 1b9319b)

Created `src/lib/tauri-safe.ts` with `safeInvoke<T>()` function that:
- Wraps Tauri's invoke with comprehensive logging
- Logs all invocations with command name and arguments: `[Tauri] Invoking {command} with args: {args}`
- Logs all successes with result: `[Tauri] Success {command}: {result}`
- Logs all errors with full error object: `[Tauri] Error {command}: {error}`
- Uses consistent `[Tauri]` prefix for easy console filtering
- Never throws silently - always logs before rethrowing
- Includes JSDoc with detailed usage examples

**Why this works:** Every IPC call now appears in browser DevTools console regardless of production/dev mode. Users and developers can screenshot console output to diagnose exactly where folder selection flow fails.

### Task 2: Add logging to ProjectPicker folder selection flow (Commit: 3a1459c)

Updated `src/components/ProjectPicker.tsx` to:
- Import safeInvoke from src/lib/tauri-safe
- Add `[DEBUG]` logging at each step:
  - `"[DEBUG] Opening folder dialog"` before openDialog call
  - `"[DEBUG] Dialog returned path: {path}"` when path selected
  - `"[DEBUG] Dialog returned no path (user cancelled)"` when cancelled
  - `"[DEBUG] onProjectSelected callback invoked successfully"` after callback
- Replace `invoke` with `safeInvoke` in handleCreateRemote for remote projects
- Add error logging with full error objects in catch blocks
- Log context for recent projects and manual path handlers

**Why this works:** Folder selection flow is now fully visible step-by-step:
1. User clicks button → logs dialog start
2. User selects folder → logs path
3. Callback fires → logs callback success
4. App proceeds to main UI (or logs error if fails)

### Task 3: Update App.tsx to use logging and handle errors visibly (Commit: 39996f5)

Updated `src/App.tsx` to:
- Import safeInvoke from src/lib/tauri-safe
- Replace `invoke` from tauri-mock with safeInvoke (ensures all calls logged)
- Add comprehensive DEBUG logging in useEffect loadSettings:
  - `"[DEBUG] App.tsx: Loading initial settings"`
  - `"[DEBUG] App.tsx: Settings loaded successfully"`
  - `"[DEBUG] App.tsx: Project path found in settings: {path}"`
  - `"[DEBUG] App.tsx: Loading project from database"`
  - `"[DEBUG] App.tsx: Project loaded successfully"`
- Add comprehensive DEBUG logging in handleProjectSelected:
  - `"[DEBUG] App.tsx: handleProjectSelected starting with path: {path}"`
  - `"[DEBUG] App.tsx: Calling get_or_create_project"`
  - `"[DEBUG] App.tsx: Project created/loaded successfully"`
  - `"[DEBUG] App.tsx: Saving settings with new project path"`
  - `"[DEBUG] App.tsx: Settings saved successfully"`
  - `"[DEBUG] App.tsx: Project selected state updated, main UI should now be visible"`
- Error catch blocks log with full context

**Why this works:** Initial app startup and project selection flows are now traceable:
- Users can open DevTools and screenshot console
- Console output shows exact sequence of IPC calls
- If flow fails, error message appears with full error object
- Developers can review console logs to identify:
  - Which IPC call failed
  - What error was returned
  - Whether failure happened in loadSettings or handleProjectSelected
