Add the following to execution.service.ts AFTER the existing executionService object:

1. **Export query key factory** (execution operations are mostly mutations; include factory for consistency):
   ```typescript
   export const executionQueryKeys = {
     all: ["executions"] as const,
     details: () => [...executionQueryKeys.all, "detail"] as const,
     detail: (executionId: number) => [...executionQueryKeys.details(), executionId] as const,
   };
   ```

2. **Add useMutation hooks for execution control** (all side-effect operations—no queries):
   - useSpawnExecutionMutation() - calls executionService.spawnExecution(taskId, config), invalidates nothing (fire-and-forget side effect), on error show toast
   - usePauseExecutionMutation() - calls executionService.pauseExecution(executionId), invalidates nothing, on error show toast
   - useResumeExecutionMutation() - calls executionService.resumeExecution(executionId), invalidates nothing, on error show toast
   - useAttachTerminalMutation() - calls executionService.attachTerminal(executionId, sessionId), invalidates nothing, on error show toast
   - useDetachTerminalMutation() - calls executionService.detachTerminal(executionId), invalidates nothing, on error show toast
   - useSendTerminalInputMutation() - calls executionService.sendTerminalInput(executionId, input), invalidates nothing, on error show toast
   - useResizeTerminalMutation() - calls executionService.resizeTerminal(executionId, cols, rows), invalidates nothing, on error show toast

Notes:
- These mutations are primarily for remote procedure calls (RPCs), not data fetching
- No complex cache invalidation needed (execution state is ephemeral)
- Each mutation should have onError callback that shows toast with error message
- Mutations return void or minimal data; focus on error handling
- No optimistic updates needed for execution control operations

Reference research doc pattern for basic mutation structure (lines 131-143 as minimal example).
