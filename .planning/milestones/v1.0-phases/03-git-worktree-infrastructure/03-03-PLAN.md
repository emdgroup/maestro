**#[tauri::command] pub async fn cleanup_worktree(...) -> Result<(), String>**

Parameters:
- project_id: i32
- worktree_id: i32
- repo_path: String (full path to git repository)
- state: State<Arc<AppState>>

Flow:

1. **Fetch worktree record** (before cleanup):
   - Query: SELECT * FROM worktrees WHERE id=? AND project_id=?
   - If not found: Return Err("Worktree not found")
   - Capture: worktree.path, worktree.branch_name

2. **Mark as 'dirty'** (transaction):
   - This ensures we can retry if process crashes mid-cleanup
   - UPDATE worktrees SET status='Dirty' WHERE id=?

3. **Execute cleanup via sidecar** (spawn async Node.js process):
   - This is where phase 3-01's git-manager functions are called
   - Use tokio::process::Command (NOT std::process::Command) because this is async function
   - Call sidecar with arguments: deleteWorktree repo_path worktree_id_str branch_name
   - Handle errors:
     * If deletion fails → mark dirty, return Err("Cleanup failed: {error}")
     * If deletion succeeds → continue to step 4

Pattern for async sidecar invocation:
```rust
let sidecar_path = get_sidecar_path()?; // Resolve sidecar binary location
let output = tokio::process::Command::new(&sidecar_path)
  .args(&["deleteWorktree", &repo_path, &worktree_id_str, &branch_name])
  .output()
  .await
  .map_err(|e| format!("Sidecar spawn failed: {}", e))?;

if !output.status.success() {
  let stderr = String::from_utf8_lossy(&output.stderr);
  return Err(format!("Sidecar error: {}", stderr));
}
```

4. **Delete from database** (atomic):
   - Start transaction
   - DELETE FROM worktrees WHERE id=? AND status='Dirty'
   - Commit
   - Log: "Cleaned up worktree {worktree_id} (branch: {branch_name})"

5. **Return success**:
   - Ok(())

Key implementation details:

- Mark as 'dirty' FIRST (durable flag for retry)
- Use tokio::process::Command for async spawning (critical - prevents deadlock)
- NOT std::process::Command (blocks thread in async context)
- If sidecar call fails, worktree stays 'dirty' for manual cleanup later
- If database delete fails, log error but don't fail (cleanup succeeded, just DB tracking)

DO NOT:
- Use std::process::Command in async function (will deadlock Tauri event loop)
- Delete from database BEFORE cleanup (if crash mid-cleanup, can't retry)
- Skip error handling—sidecar may fail
- Call git commands directly from Rust (go through sidecar for consistency)

**#[tauri::command] pub async fn recover_dirty_worktrees(...) -> Result<Vec<i32>, String>**

Purpose: Called on app startup or periodically to retry cleanup of worktrees stuck in 'dirty' state.

Parameters:
- project_id: i32
- repo_path: String
- state: State<Arc<AppState>>

Logic:
1. Query: SELECT id, branch_name, path FROM worktrees WHERE project_id=? AND status='Dirty'
2. For each dirty worktree:
   a. Attempt cleanup via sidecar (same pattern as cleanup_worktree, use tokio::process::Command)
   b. On success: DELETE FROM worktrees WHERE id=?
   c. On failure: Log error, keep in 'Dirty' state (will retry next time)
3. Return list of successfully recovered worktree IDs (for logging)

Pattern:
```rust
pub async fn recover_dirty_worktrees(...) -> Result<Vec<i32>, String> {
  // ...
  let mut recovered_ids = vec![];
  for dirty_wt in dirty_worktrees {
    match cleanup_via_sidecar(&repo_path, &dirty_wt).await {
      Ok(_) => {
        // Delete from DB
        recovered_ids.push(dirty_wt.id);
      }
      Err(e) => {
        // Log error, leave in dirty state
        eprintln!("Failed to recover worktree {}: {}", dirty_wt.id, e);
      }
    }
  }
  Ok(recovered_ids)
}
```

Integration:
- Will be called from App.tsx useEffect on project load (Phase 2 integration point)
- Can also be invoked manually by user if stuck worktrees accumulate
- Add documentation noting where this should be called (see Task 3)

DO NOT:
- Force delete from database if cleanup fails
- Skip logging failures
- Use blocking I/O—use async/await with tokio
