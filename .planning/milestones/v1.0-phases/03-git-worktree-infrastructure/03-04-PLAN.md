**#[tauri::command] pub fn initialize_worktree_pool(...) -> Result<PoolStatus, String>**

Parameters:
- project_id: i32
- repo_path: String (full path to git repository)
- pool_size: Option<i32> (default: 3, override for testing)
- state: State<Arc<AppState>>

Logic:

1. **Check existing pool**:
   - Query: SELECT COUNT(*) FROM worktrees WHERE project_id=? AND status='available'
   - If count >= pool_size, return Ok(current_pool_status) (already initialized)
   - Otherwise, continue to step 2

2. **Create missing worktrees** (within transaction):
   - Calculate how many to create: needed = pool_size - current_count
   - For i = 1 to needed:
     a. Generate worktree_id: wt-{current_count + i} (zero-padded, e.g., wt-001, wt-002, etc.)
     b. Generate branch: pool/reserved-{uuid} or pool/init-{timestamp} (temporary, will be replaced on task lease)
     c. Generate path: .worktree-pool/{worktree_id}
     d. Insert row: INSERT INTO worktrees (project_id, branch_name, path, status, created_at) VALUES (..., 'available', ...)
   - Commit transaction

3. **Return pool status**:
   - Call get_pool_status to return current pool state
   - Indicates: total created, available ready, utilization

Key patterns:
- Use transaction for atomicity (prevents concurrent duplicate creation)
- Mark all created worktrees as 'available' (not 'leased')
- Timestamps: chrono::Utc::now().to_rfc3339()
- Do NOT actually create git worktrees yet—just database records
  - Actual git worktree creation happens when worktree is leased (lazy creation)
  - This approach avoids disk I/O on project open, keeps startup fast

Design rationale:
- Pre-create database entries only, not actual git worktrees
- Lazy git creation on first lease (when needed for task execution)
- Provides instant "available" worktrees without slow disk I/O at startup
- Actual git worktree branch gets created when lease happens (via git-manager)

Const:
- DEFAULT_POOL_SIZE = 3 (pre-create on open)
- MAX_POOL_SIZE = 5 (dynamic expansion limit from Phase 3-02)

DO NOT:
- Create actual git worktrees at this stage (too slow, unnecessary)
- Skip transaction (concurrent project opens could create duplicates)
- Hard-code pool size (use const and allow override for testing)
