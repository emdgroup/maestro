Output: Error type definitions, database layer for error storage, and backend integration that detects failures, categorizes them, and pauses execution with suggestions.

<task type="auto">
  <name>Task 2: Implement error detection and categorization logic in Rust handlers</name>
  <files>src-tauri/src/ipc/handlers.rs</files>
  <action>
Implement error detection in spawn_agent_execution background task (after PTY session completes):

1. Monitor PTY session for process exit code and stderr output
2. When process exits (code != 0), parse stderr for patterns:
   - Compilation errors: detect patterns like "error TS", "error:", "SyntaxError", "ReferenceError"
   - Missing dependencies: detect "not found", "Cannot find module", "npm ERR!", "package.json"
   - Runtime errors: detect "Error:", "Exception", "Panic", "segmentation fault"
   - Timeouts: if execution exceeds 5 minutes (claude's discretion on timeout), mark as timeout
   - Process crash: exit code < 0 or signal termination

3. Generate error suggestions based on type:
   - CompilationError → ["Run: npm install", "Check syntax in source files"]
   - MissingDependency → ["Run: npm install", "Check package.json dependencies"]
   - RuntimeError → ["Check task acceptance criteria", "Review error in terminal history"]
   - Timeout → ["Task may be long-running or stuck; check terminal for activity"]
   - ProcessCrash → ["Check system resources", "Review agent logs"]
   - Unknown → ["Review full terminal output", "Check error details"]

4. Create ErrorEvent struct with categorization and suggestions

5. Call append_error_event (implemented in next task) to store error in database

6. Update execution log status to Failed

7. Mark task status as Failed (or Paused - claude's discretion; recommend Failed to align with workflow states)

Note: Implement auto-retry logic (max 3 retries) for retry-able errors only:
- Retry-able: network timeouts, temporary file locks, transient process failures
- Fatal: compilation errors, missing dependencies, syntax errors
- Use heuristic: if error_type in [Timeout, ProcessCrash], auto-retry; otherwise pause immediately
- For auto-retry: increment retry_count in execution log, re-spawn process with same command, don't notify user until final failure
  </action>

- Grep for "error_type" in handlers.rs confirms error categorization logic exists
- Grep for "auto-retry" confirms retry loop for retry-able errors
- Grep for "mark_failed" confirms failed status is set
  </verify>
  <done>
Error detection logic implemented. Process exit codes monitored. Stderr parsed for error patterns. Error types categorized with suggestions. Auto-retry logic for transient failures. Task status updated to Failed when error occurs.
  </done>
</task>
