# Phase 44: DB Schema + ACP IPC Handlers - Research

**Researched:** 2026-04-20
**Domain:** SQLite schema migration (rusqlite), Tauri IPC command refinement, tokio periodic flush
**Confidence:** HIGH

## Summary

Phase 44 has two separable concerns that must land together: (1) a database schema bump from v10 to
v11 adding three new columns to `execution_logs`, and (2) completing the IPC surface so the frontend
can drive ACP sessions with named, purpose-specific commands rather than the generic `send_to_acp_session`
dispatch introduced in Phase 43.

The schema migration is straightforward. The existing migration path in `db/schema.rs` uses a
drop-and-recreate strategy (no production data to preserve), so bumping `SCHEMA_VERSION` from 10 to
11, adding three `ALTER TABLE` statements for the upgrade path, and updating `SCHEMA_V10` → `SCHEMA_V11`
with the three new columns is the full scope. The new columns are:
`execution_mode TEXT NOT NULL DEFAULT 'pty'` (guards backward compat for all existing PTY rows),
`agent_id TEXT` (nullable — only set for ACP sessions), and
`structured_output TEXT` (nullable — accumulates JSON-serialized `SessionUpdate` payloads).

The IPC surface in Phase 43 implemented three generic commands: `start_acp_session`,
`send_to_acp_session`, `cancel_acp_session`. The Phase 44 success criteria mention four *specific*
IPC commands: `spawn_acp_session`, `send_acp_prompt`, `respond_acp_permission`, and cancel.
Reading the requirements carefully: PERSIST-02 says "spawn ACP session via IPC", PERSIST-03 says
"send prompt", PERSIST-04 says "respond to permission request", PERSIST-05 says "cancel".
The PERM-02 requirement explicitly names `respond_acp_permission`.
The plan must either (a) rename the Phase 43 commands to the canonical names, or (b) add new
dedicated commands and deprecate/remove the generic one. Option (a) is cleaner — the frontend
bindings are regenerated by `pnpm tauri:gen` and the frontend has not yet consumed these commands.

Phase 43 also left a TODO: `start_acp_session` inserts the execution_log row without writing
`execution_mode='acp'` or `agent_id` because those columns do not yet exist. Phase 44 adds the
columns and updates the INSERT to populate them.

The structured output flush (PERSIST-06) requires the background reader task in `acp/manager.rs`
to accumulate `SessionUpdate` payloads and periodically write them to
`execution_logs.structured_output`. The existing `tokio::time::interval` pattern for periodic
work is already present in the SSH heartbeat (`ssh/session.rs`). A 10-second flush interval is
appropriate — frequent enough for dead-session replay fidelity, rare enough to avoid DB write
saturation.

**Primary recommendation:** Bump schema to v11 with three new columns; rename IPC commands to
canonical names (`spawn_acp_session`, `send_acp_prompt`, `respond_acp_permission`,
`cancel_acp_session`); update the INSERT in `spawn_acp_session` to set `execution_mode='acp'` and
`agent_id`; add a `tokio::time::interval(10s)` select arm to the reader task that flushes
accumulated `structured_output` to DB.

### Pattern 1: Schema Migration (Drop-and-Recreate)

**What:** Bump `SCHEMA_VERSION`, rename `SCHEMA_V10` constant to `SCHEMA_V11`, add three columns to the `execution_logs` CREATE TABLE, update the DROP list if needed.
**When to use:** Version < SCHEMA_VERSION triggers full drop + recreate. No in-place ALTER needed because there is no production data to preserve (dev environment only).

// Source: src-tauri/src/db/schema.rs (existing pattern)

**Key constraint:** `execution_mode TEXT NOT NULL DEFAULT 'pty'` ensures all existing PTY rows
get a default value and the column is non-nullable without a migration step for existing data.

### Pattern 2: Periodic Flush in tokio::select! Loop

**What:** Add a `tokio::time::interval` arm to the existing `tokio::select!` reader loop.
Accumulate `SessionUpdate` payloads in a `Vec<serde_json::Value>` local variable. Every 10 seconds,
serialize and write to `execution_logs.structured_output`.

```rust
// Source: ASSUMED pattern based on ssh/session.rs heartbeat + acp/manager.rs reader
use tokio::time::{interval, Duration};

let mut flush_interval = interval(Duration::from_secs(10));
let mut structured_updates: Vec<serde_json::Value> = Vec::new();

loop {
    tokio::select! {
        biased;
        _ = &mut cancel_rx => break,
        _ = flush_interval.tick() => {
            if !structured_updates.is_empty() {
                let json = serde_json::to_string(&structured_updates)
                    .unwrap_or_default();
                if let Ok(conn) = app_state.db.lock() {
                    let _ = conn.execute(
                        "UPDATE execution_logs SET structured_output = ?1 WHERE id = ?2",
                        rusqlite::params![&json, log_id],
                    );
                }
                // Do NOT clear structured_updates — overwrite on every flush
                // so the full accumulated list is always in DB
            }
        }
        result = read_message(&mut stdout_reader) => {
            match result {
                Ok(MaestroRpcMessage::Response(ServerResponse::SessionUpdate(upd))) => {
                    structured_updates.push(upd.payload.clone());
                    let _ = app_handle.emit(
                        &format!("acp://session-update/{}", log_id),
                        &upd.payload,
                    );
                }
                // ... other arms unchanged
            }
        }
    }
}

// Final flush on exit
if !structured_updates.is_empty() {
    let json = serde_json::to_string(&structured_updates).unwrap_or_default();
    if let Ok(conn) = app_state.db.lock() {
        let _ = conn.execute(
            "UPDATE execution_logs SET structured_output = ?1 WHERE id = ?2",
            rusqlite::params![&json, log_id],
        );
    }
}
```

**Critical:** The `biased;` qualifier in `tokio::select!` must remain — it ensures cancellation is
checked before reads. The interval arm must come after cancel but before read_message to prevent
a chatty server from starving the flush timer. [ASSUMED — confirm biased ordering is correct for
this use case; the existing code already uses `biased;`]

### Pattern 3: Dedicated IPC Commands (Split from Generic Dispatcher)

**What:** Replace `send_to_acp_session` (which dispatches on `message_type: String`) with two
purpose-specific commands. This gives the frontend type-safe bindings and removes runtime string
dispatch.

```rust
// Source: Derived from existing acp_handlers.rs send_to_acp_session pattern

#[tauri::command]
#[specta::specta]
pub async fn send_acp_prompt(
    app_state: State<'_, Arc<AppState>>,
    log_id: i32,
    content: String,
) -> Result<(), String> {
    let session_id = format!("session-{}", log_id);
    let msg = MaestroRpcMessage::Request(ServerRequest::Prompt(PromptRequest {
        session_id,
        content,
    }));
    crate::acp::write_to_acp_session(&app_state, log_id, &msg).await
}

#[tauri::command]
#[specta::specta]
pub async fn respond_acp_permission(
    app_state: State<'_, Arc<AppState>>,
    log_id: i32,
    request_id: String,
    allowed: bool,
) -> Result<(), String> {
    let session_id = format!("session-{}", log_id);
    let msg = MaestroRpcMessage::Request(ServerRequest::PermitResponse(PermissionResponse {
        session_id,
        request_id,
        allowed,
    }));
    crate::acp::write_to_acp_session(&app_state, log_id, &msg).await
}
```

### Anti-Patterns to Avoid

- **Clearing `structured_updates` after each flush:** The column should always contain the full
  accumulated list for replay. Overwrite every flush, never truncate mid-session.
- **Holding DB lock across `tokio::select!` await:** Never hold `std::sync::MutexGuard<Connection>`
  across an `.await`. Acquire lock, write, drop immediately — established pattern in all handlers.
- **Forgetting `tauri:gen` after renaming IPC fns:** The TypeScript bindings in
  `src/types/bindings.ts` are auto-generated. Any rename of `#[tauri::command]` fns or parameter
  changes requires `pnpm tauri:gen` to be run as the final task in the plan.
- **Not registering new commands in `create_builder()`:** `lib.rs` has an explicit
  `collect_commands![]` list. Every new or renamed command must be registered there.

## Common Pitfalls

### Pitfall 1: Forgetting to Register Renamed Commands in lib.rs

**What goes wrong:** `cargo check` passes (fn is valid Rust) but the IPC command is silently absent
from the frontend. `pnpm tauri:gen` succeeds but generates stubs for functions that don't exist in
the collect_commands! list — or fails to generate stubs for new names.

**Why it happens:** `lib.rs` has an explicit allowlist in `create_builder()`. Renaming a fn in
`acp_handlers.rs` does not automatically update the list.

**How to avoid:** After every rename or addition, update the `collect_commands![]` block in `lib.rs`
and run `pnpm tauri:gen`.

**Warning signs:** Frontend TypeScript calls an old command name; runtime panics with
"Command not found" error.

### Pitfall 2: `execution_mode` DEFAULT Does Not Apply to In-Progress Writes

**What goes wrong:** The DROP-and-recreate migration wipes all rows, so the DEFAULT applies cleanly
to the empty table. BUT if somehow a migration path is used that does `ALTER TABLE ... ADD COLUMN`
instead, the DEFAULT must be explicit in the ALTER statement, not just in the CREATE TABLE.

**Why it happens:** SQLite `ADD COLUMN` without a DEFAULT clause inserts NULL for existing rows
(prior to SQLite 3.37), not the column default.

**How to avoid:** Use the existing drop-and-recreate strategy, not ALTER TABLE. The comment in
`schema.rs` states "no production data to preserve" — stay on this path.

**Warning signs:** Existing execution_log rows have NULL for execution_mode after migration.

### Pitfall 3: `biased;` Ordering in tokio::select! with Interval

**What goes wrong:** Without `biased;`, tokio::select! polls branches in a random order. The
cancel branch may lose to the read_message branch, causing the reader task to hang during shutdown.

**Why it happens:** Default select! is fair (pseudorandom polling). The existing code uses `biased;`
to prioritize cancellation — the interval arm must be positioned correctly relative to cancel_rx
and read_message.

**How to avoid:** Keep `biased;`. Order branches: cancel_rx first, flush interval second, read_message
last. This ensures cancellation is always checked first, flush doesn't block reads.

**Warning signs:** Session cleanup takes up to 10 seconds instead of being immediate; reader task
lingers after cancel_acp_session returns.

### Pitfall 4: list_executions_with_task_info SQL Query Not Updated

**What goes wrong:** Phase 44 adds `execution_mode` and `agent_id` to the schema, but the
`list_executions_with_task_info` SELECT query in `execution_handlers.rs` does not include them.
The frontend receives `ExecutionWithTask` structs without these fields — future phases (46, 47)
that need to distinguish ACP vs PTY sessions by `execution_mode` will have to re-patch this.

**Why it happens:** The query was written before v11 schema existed.

**How to avoid:** Add `el.execution_mode, el.agent_id` to the SELECT list and extend the
`ExecutionWithTask` struct. Then `pnpm tauri:gen` regenerates the TypeScript type.

**Warning signs:** Phase 46 Agent Selector cannot distinguish ACP sessions from PTY sessions.

### Pitfall 5: DB Lock Held Across Flush Await

**What goes wrong:** If the DB lock is acquired inside the select! arm and not immediately dropped
before the next iteration, it crosses an implicit yield point.

**Why it happens:** Rust borrow checker will not catch this — `std::sync::MutexGuard` is not
`Send`, but `tokio::spawn(async move { ... })` moves it.

**How to avoid:** Use scoped blocks `{ let conn = ...; conn.execute(...)?; }` so the guard is
dropped immediately after the execute. This is the established pattern in every other handler.
