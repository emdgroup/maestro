1. **#[tauri::command] pub async fn lease_worktree(...) -> Result<Worktree, String>**

   Parameters:
   - project_id: i32
   - task_id: i32
   - repo_path: String (path to project repository on disk)
   - state: State<Arc<AppState>>
   - app_handle: AppHandle (from tauri context, for sidecar communication)

   Logic (with transaction):
   a. Lock database mutex
   b. Start transaction
   c. Query: SELECT * FROM worktrees WHERE project_id=? AND status='Available' LIMIT 1
   d. If found:
      - Update status to 'Leased', set leased_at = NOW()
      - Commit transaction
      - Return Worktree (use existing worktree already on disk)
   e. If not found:
      - Count total worktrees: SELECT COUNT(*) FROM worktrees WHERE project_id=?
      - If count < 5 (max pool size):
        * Generate worktree_id: wt-001, wt-002, ... (zero-padded)
        * Generate branch: pool/agent-task-{task_id}
        * Generate path: .worktree-pool/{worktree_id}
        * Insert new row: INSERT INTO worktrees (project_id, branch_name, path, status, created_at) VALUES (...)
        * Update to 'Leased' immediately
        * Commit transaction
        * **CRITICAL: After transaction commits, BEFORE returning:**
          - Use tokio::process::Command to spawn sidecar:
            ```rust
            let output = tokio::process::Command::new("sidecar")
              .args(&["createWorktree", "--repo", repo_path, "--worktree-id", worktree_id, "--task-id", task_id.to_string(), "--branch", branch_name])
              .output()
              .await
              .map_err(|e| format!("Sidecar spawn failed: {}", e))?;

            if !output.status.success() {
              let stderr = String::from_utf8_lossy(&output.stderr);
              return Err(format!("Sidecar createWorktree failed: {}", stderr));
            }
            ```
          - If sidecar fails, mark worktree as 'Dirty' and return Err with reason
          - Only return Worktree if sidecar succeeds
      - Else:
        * Return Err("Pool exhausted, max 5 worktrees reached")

   Key points:
   - Use transaction() for atomicity (prevents two threads creating at same path)
   - chrono::Utc::now().to_rfc3339() for timestamps
   - Error messages describe why (exhausted, lock failed, query failed, sidecar failed)
   - **CRITICAL: Create database record BEFORE calling sidecar (transaction must succeed first)**
   - **CRITICAL: Sidecar invocation uses tokio::process::Command (async), NOT std::process::Command (blocking)**
   - If sidecar fails after database commit, mark worktree as "Dirty" for Phase 3-03 cleanup logic to handle
   - Do NOT return worktree on sidecar failure (prevents returning non-existent directory)

2. **#[tauri::command] pub fn return_worktree(...) -> Result<(), String>**

   Parameters:
   - worktree_id: i32
   - state: State<Arc<AppState>>

   Logic:
   - Lock database mutex
   - Update: UPDATE worktrees SET status='Available', returned_at=NOW() WHERE id=?
   - Return Ok(()) or Err(...)
   - Do NOT delete or touch the worktree on disk (Phase 3-03 cleanup handles deletion)

Implementation reference:
- Phase 3 research doc lines 384-450 (Rust transaction pattern)
- Phase 3 research doc lines 345-376 (sidecar createWorktree TypeScript, adapt to Rust invocation)

DO NOT:
- Skip transaction—race condition will corrupt pool state
- Hardcode pool size (use const POOL_MAX_SIZE = 5 at module level)
- Return user-facing errors without context
- Forget to unlock mutex (use map_err patterns)
- Call sidecar BEFORE transaction commits (order is critical)
- Use std::process::Command—use tokio::process::Command for async
- Return success if sidecar fails (always check exit status)
- Attempt filesystem operations directly (all git worktree creation via sidecar)

**#[tauri::command] pub fn get_pool_status(...) -> Result<PoolStatus, String>**

Parameters:

- state: State<Arc<AppState>>

- Lock database mutex
